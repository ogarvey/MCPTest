- Korokke BIND_output asset filenames are often padded before the extension, e.g. ALL     .TIM, BPARTS  .TIM, BFACE01 .TIM.
- When resolving sibling TIMs, match by trimmed stem plus extension instead of exact filename.
- If a palette extraction falls back to synthetic unexpectedly, check padded archive filenames first.
- ANC files do not store an absolute palette index. Runtime palette selection combines two ANC fields with entry state: frame descriptor word4 low byte selects a per-entry CLUT base byte, and each tile's third byte is a relative attribute offset inside that base.
- RegisterAncSlotRange initializes the per-entry CLUT base byte table at DAT_8008d2a0..DAT_8008d2ab from resourceId; RenderAssetEntries reads byte [8 + PageSlot] from that table and passes it as QueueAncTileGrid param_7.
- UploadAncTextureData uploads the full ANC atlas VRAM words at x=((resourceId&0xF)<<6), y=((resourceId>>4)<<8), width=texturePageColumns*0x40, height=0x100.
- QueueAncTileGrid samples CLUTs from those same uploaded atlas rows: clutY is the bottom-of-bank row 0xFF minus (attribute>>2), and clutX selects one of the 16-color blocks packed into the atlas upload width.
- For battle ANC extraction, prefer palette groups decoded from the ANC atlas upload itself instead of heuristic ALL.TIM/BPARTS/BFACE staging. The old external-TIM path produced the purple corrupted BCHARA01 frame_0000; the embedded-atlas path produced correct-looking BCHARA00/BCHARA01/ICHARA00 sprites.
- ANC sequence alignment is not recoverable from frame crop size alone. Runtime notes show StepAssetEntryScript updates per-entry offsets before RenderAssetEntries draws, and BCHARA00 sequence_010 changes FB/FC command values exactly where the animation switches between 56x56 and 48x56 frames.
- Verified in Ghidra on SLPM_872.35: StepAssetEntryScript opcode 0xFC writes the per-entry X offset at DAT_8008d304, opcode 0xFB writes the per-entry Y offset at DAT_8008d306, and RenderAssetEntries adds those to the draw position passed into QueueAncTileGrid.
- QueueAncTileGrid treats the incoming X/Y as the top-left corner of the composed sprite box; width and height extend positively from there, and F9 low bits flip texture mapping in-place without changing that box.
- FCHARA follow-on ANC packs are support/effect assets, not clean standalone pose sheets. FCHARA08 contains many descriptor windows whose ANC cells are literally all tileCode 0x0000 (true blank frames), plus sparse and multi-cluster windows; scans found no out-of-bounds atlas tile references, so the blank/cut-off look is in the frame definitions themselves rather than a missing external atlas page.
- BCHARA08 vs FCHARA08: the traced SetupBattleScene branch that loads FCHARA08 does not also load BCHARA08; it swaps to an FCHARA-only set (FCHARA08 plus FCHARA11/FCHARA13/FCHARA14), while the normal branch loads BCHARAxx/ICHARAxx instead. RenderAssetEntries resolves frame descriptors from the current entry's registered ancData pointer, and StepAssetEntryScript only changes frame id / transforms, so blank FCHARA frames cannot fall through to BCHARA at render time. Exact rendered-frame compare found 26 byte-identical shared frames (0000-0013, 0016-0018, 0037, 0070-0071, 0091-0096), 12 visible F-only frames (0022-0024, 0028-0030, 0097-0102), and 65 true blank F frames, which fits a script-compatible alternate/support pack rather than cross-file frame reuse.
- Direct-quad runtime semantics: QueueSpriteQuad uses the full 16-bit frame word0 as U, byte-sized word1 as V, and if U > 255 it folds the overflow into tpage base instead of treating word0 high-byte as a palette index. For ANM, RegisterAnmSlotRange/UploadAnmPagesToVram confirm type 0 is 4bpp 256x256, type 1 is 8bpp 256x256, and type 3 is runtime-special 8bpp 128x256 with its own embedded 0x100-color palette row. For ANN, UploadAnnTextureData uploads only texture pixels and RenderAssetEntries derives the final banked CLUT/tpage coordinates from the entry resource byte plus frame word4 high-byte; the old extractor heuristic using only (PageVariantHighByte & 0x0F) was too simple.
- Concrete ANN CLUT sources are scene-side, not file-side. Continue mode creates a scratch TIM with CreateBlankTimResource and uploads it before loading CONTINUE.ANN. Battle mode uploads TIMs like BPARTS.TIM, BFAC??E.TIM, BFAC??R.TIM, optional FUSEN.TIM, and FFACE.TIM before registering GIMICK.ANN / BPART2.ANN / EFFECT.ANN, so ANN color recovery must trace those earlier TIM/VRAM staging calls.
- QueueSpriteQuad packs the final CLUT exactly like PSX GPU `GetClut(x,y)`: `(clutX >> 4) + clutY * 64`. Static xrefs in `SLPM_872.35` show only two direct ANN registration/upload callers: `SetupBattleScene` and `FUN_8003be70` for the continue screen. If `F00.ANN` is used, it is being reached through a more indirect package/script path rather than a literal direct ANN registration call in this executable.
- `FUN_80029a1c` does not build `.ANN` names in the traced ADV/MES interpreter path. The dynamic basename cases there append only `.MP`, `.TIM`, and `.MES`, with the map case also calling `LoadPackageAndMpk`. Current literal `.ANN` string ownership in the executable is limited to battle (`GIMICK.ANN`, `BPART2.ANN`, `EFFECT.ANN`) and continue screen (`CONTINUE.ANN`).
- Focused `F00.ANN` extractor/debug pass: `textureOffset = 0x920`, so `0x4C0..0x91F` is all pre-texture data; the strongest ranked candidate windows are around `0x640..0x720` and do not overlap the 15 used frame descriptors, but the strongest raw block at `0x700` still consists entirely of low-byte words (`<= 0x00FF`), which is weak evidence for a normal internal PSX CLUT.
- ANM page type 03 refinement: runtime uploads type 03 blocks with the same 0x40-word width as a 128x256 8bpp page, then RenderAssetEntries coerces page mode 3 to QueueSpriteQuad textureMode 1 before drawing. The previous extractor treated type 03 as 4bpp 256x256, which changed CUT00/BPART4 outputs incorrectly; decoding type 03 as 8bpp 128x256 matches the runtime geometry and materially changes the exported pages/frames.
